# Reference rules

Reference is a dictionary: austere, authoritative description of the machinery, consulted mid-work. The reader came for a fact and wants to leave with it in seconds.

## Structure

- Mirror the structure of the product: organize the doc the way the code is organized, so readers navigate by what they already know.
- One format, applied uniformly: every entry gets the same heading pattern and the same field order (signature, parameters with types and defaults, return value, errors raised, constraints).
- State facts in neutral tone: what it is, what it accepts, what it returns, what it guarantees.
- Examples are allowed to *illustrate* an entry — a line or two showing shape, not a walkthrough.

## The purity rule

Describe; never instruct or persuade. Steps toward a goal live in how-to guides; why-it-is-this-way lives in explanation — link to both.

## Completion

The reference is done when every item in the agreed scope has an entry, every entry follows the same format, and every signature, type, and default is verified against the actual code.
