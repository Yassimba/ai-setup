# Explanation rules

Explanation is a discussion: understanding-oriented reading for someone stepping back from the work. It can be read in an armchair, away from the keyboard.

## Structure

- Frame the topic so the title works with "About" in front: "About caching", "Why we chose event sourcing".
- Provide context: history, constraints, design decisions, alternatives considered and why they lost.
- Make connections: relate the topic to adjacent parts of the system and to ideas the audience already holds.
- Multiple viewpoints, trade-offs, and "it depends" are all admissible; opinion is welcome when marked as such.

## The purity rule

Keep the reader thinking, not doing. When steps start creeping in, link to the how-to guide; when raw facts pile up, link to reference.

## Completion

The explanation is done when a reader matching the agreed audience can answer "why is it this way?" for the topic — and every factual claim about the system is verified against the actual code or the sources provided.
