# How-to guide rules

A how-to guide is a recipe: directions for a competent reader to achieve a real-world goal. The reader is mid-work and knows their tools — teaching belongs in tutorials.

## Structure

- Title names the goal: "How to <achieve X>". If the title needs "and", it is two guides.
- Open with any preconditions (versions, access, prior setup), then go straight into an ordered sequence of actions.
- Address real-world messiness: name the forks that matter ("if you use Y, do Z instead") rather than pretending one clean path.
- Cover what the goal needs and stop — completeness for its own sake belongs in reference.

## The purity rule

Every line serves the goal. Rationale and background live in explanation docs; exhaustive option lists live in reference — link to both rather than detouring.

## Completion

The guide is done when a competent reader can go from the stated preconditions to the achieved goal without leaving the page except through links you chose, with every command and snippet verified against the actual code.
