# Tutorial rules

A tutorial is a lesson: the reader learns by doing, under your guidance. You are the teacher and take full responsibility for the learner's success — if a step can fail, that is your problem to prevent.

## Structure

- Open by stating what the learner will build and show the finished result up front.
- Order steps so a visible result appears early and after every step: "run this — you should see …". Each checkpoint lets the learner confirm they are on track.
- Give exactly one way to do each thing: the concrete path you have verified works. Alternatives and options belong in how-to guides.
- Use "we" throughout — you are doing it together.
- Minimum viable explanation: one sentence of "why" where it keeps the learner oriented; when more understanding-pressure builds, link to an explanation doc and move on.

## The purity rule

Keep the learner *doing*. Every paragraph either instructs an action or confirms its result. Abstraction, completeness, and background reading all live in other quadrants — link to them.

## Completion

The tutorial is done when a newcomer matching the agreed audience can follow it start to finish and end with the promised working result, with every command and snippet verified against the actual code.
